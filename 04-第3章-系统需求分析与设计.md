第 3 章 系统需求分析与设计

3.1 可行性分析

医疗门诊业务涵盖预约、排班、诊疗记录与处方等高频流程，系统建设涉及前后端开发、数据存储与权限管控等多方面投入，因此在立项阶段需进行可行性评估。本系统从技术、经济与操作三个维度展开分析，综合判断技术路线成熟度、实施成本与推广难度，明确项目风险边界并为后续需求分析与系统设计提供依据。

3.1.1 技术可行性

系统技术路线基于 Spring Boot 3.2、MyBatis-Plus、MySQL 与 Vue 3 等成熟开源组件，能够支撑门诊预约、排班与处方等业务的稳定运行。后端通过 Spring Security 与 JWT 实现登录鉴权，配合分层架构与统一接口规范，便于扩展与维护；前端采用 Vite 与 Element Plus 提升构建效率与界面一致性。现有技术栈社区成熟、文档完备，开发人员具备相关经验，技术实现风险可控。

3.1.2 经济可行性

系统选用开源框架与通用数据库，无需支付商业授权费用，部署环境可采用通用服务器或云主机，硬件投入可控。统一平台替代分散登记与手工流转后，可减少重复录入、纸质归档与跨科室核对的人力成本，并降低因信息不一致导致的返工与服务延迟。综合建设与运维投入，系统具备可接受的成本结构与投入产出比。

3.1.3 操作可行性

系统面向医护与管理人员提供表单化操作界面，预约、排班、就诊记录与处方维护流程清晰，关键字段具备校验与状态提示，降低误操作概率。前端采用浏览器访问方式，科室与岗位可在同一入口完成业务处理，培训成本与迁移成本较低。结合角色权限控制与日志留痕，系统具备可推广性与长期运行基础。

3.2 系统功能需求

3.2.1 用户主要功能

用户侧以患者为核心，需支持注册登录、个人信息维护、预约挂号与时间段选择，能够查询医生排班与科室信息，并在就诊完成后查看诊疗记录与处方明细；同时提供取消或查询预约状态、查看历史就诊与用药记录等功能，以保障就诊流程连续性与信息可追溯性。

3.2.2 管理员主要功能

管理员侧承担账号权限与基础数据维护任务，需要完成用户、科室、医生与患者档案的增删改查，维护排班与预约状态并核查就诊记录，确保流程闭环；同时需管理药品与药品分类、处方与处方明细等数据，并提供统计与查询能力，以支撑门诊业务有序运行与合规管理。

3.3 非功能性需求分析

3.3.1 性能需求

系统需在门诊高峰期同时处理预约申请、排班查询与就诊记录录入等并发请求，核心页面的响应时间应保持在可接受范围内，列表查询需支持分页与条件过滤以降低数据库压力。预约与排班状态变更需及时同步，避免出现号源冲突与数据滞后，确保业务链路在高负载下保持稳定。

3.3.2 安全性需求

系统涉及患者身份、诊疗记录与处方等敏感信息，需通过登录鉴权、角色权限控制与操作审计保障数据访问边界；接口层应校验请求合法性，防止越权与非法操作。数据库层需要定期备份与恢复机制，并对关键字段设置非空与唯一约束，降低数据篡改与丢失风险。

3.3.3 易用性与可维护性需求

系统界面应保持信息层级清晰，表单输入与列表查询提供明确提示与错误反馈，减少医护人员的学习负担；常用功能支持快捷查询与状态筛选，提升业务处理效率。后端采用 Controller、Service、Mapper 分层与模块化设计，便于后续功能迭代与问题定位；前端组件化与统一样式规范有利于长期维护与扩展。
